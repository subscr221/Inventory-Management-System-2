# Story 1.3: Statutory Edit Log

Status: review

## Story

As a statutory auditor,
I want every mutating API operation recorded in a tamper-proof, non-disableable edit log with user, role, location, timestamp, and trace_id,
So that I can produce a complete audit trail for any transaction without requesting extracts, satisfying the Companies (Accounts) Rules 2014 audit-trail proviso.

## Acceptance Criteria

1. **Given** an authenticated user submits any mutating API request (POST, PUT, PATCH, DELETE)
   **When** the request is processed
   **Then** an entry appears in the edit log containing `trace_id`, `user_id`, `role`, `location_id`, `timestamp` UTC, `endpoint`, `method`, and the resulting `event_id`; the entry is written atomically with the event

2. **Given** any role (including system administrator) attempts to delete or modify an existing edit log entry via any API route or direct DB connection
   **When** the operation is attempted
   **Then** the operation is rejected and the rejection attempt is itself logged with `error_code: "AUDIT_LOG_TAMPER_ATTEMPT"`

3. **Given** an auditor calls `GET /api/v1/audit/log` with a date range and optional user filter
   **When** the response is returned
   **Then** entries appear in append order with no sequence gaps and include a range digest for integrity verification

4. **Given** a configuration flag attempts to disable the edit log
   **When** any subsequent mutating request is made
   **Then** the request is blocked with `error_code: "AUDIT_LOG_DISABLED"` - no mutating operation proceeds without the log active

5. **Given** edit-log entries from prior financial years
   **When** retention is evaluated or an early purge is attempted
   **Then** every entry remains retrievable for at least 8 financial years per books-retention (FR-AC-13) - online, or restored from the permanent S3 Glacier archive to queryable within 48 hours (NFR-S-05) - and no deletion path exists inside the retention window; any early-deletion attempt is rejected and logged with `error_code: "AUDIT_LOG_TAMPER_ATTEMPT"`

### Audit Log Data Model

```sql
-- read/projections/audit_log.sql
CREATE TABLE IF NOT EXISTS audit_log (
  log_id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  trace_id          TEXT NOT NULL,              -- per-request trace_id from router (server-generated, not from event envelope)
  user_id           UUID NOT NULL,              -- authenticated caller, server-side bound
  role              TEXT NOT NULL,              -- primary role of the caller
  location_id       TEXT NOT NULL,              -- location where the operation occurred (UUID string or '*')
  timestamp         TIMESTAMPTZ NOT NULL DEFAULT now(),  -- server timestamp UTC
  endpoint          TEXT NOT NULL,              -- request path (e.g., '/api/v1/events')
  method            TEXT NOT NULL CHECK (method IN ('GET','POST','PUT','PATCH','DELETE')),
  event_id          UUID,                       -- the resulting event_id (NULL if write failed before event creation)
  http_status       INT,                        -- HTTP response code (200, 400, 401, 403, 423, etc.)
  error_code        TEXT,                       -- error code if the operation failed (NULL if successful)
  details           JSONB,                      -- flexible audit metadata (request size, response size, etc.)
  archived          BOOLEAN DEFAULT false,      -- true if exported to Glacier/archive
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE IF NOT EXISTS audit_log_archive (
  archive_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  original_log_id   UUID NOT NULL,              -- foreign key to audit_log (log_id)
  archive_path      TEXT NOT NULL,              -- S3 path or local file path
  archived_at       TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE IF NOT EXISTS audit_log_tamper_attempt_log (
  attempt_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  timestamp         TIMESTAMPTZ NOT NULL DEFAULT now(),
  user_id           UUID,                       -- NULL if the attempt came from an unauthenticated source
  role              TEXT,                       -- NULL if unauthenticated
  location_id       TEXT,                       -- NULL if unauthenticated
  endpoint          TEXT,                       -- the endpoint being attacked
  method            TEXT,                       -- HTTP method of the attack
  error_code        TEXT NOT NULL,              -- AUDIT_LOG_TAMPER_ATTEMPT, AUDIT_LOG_DISABLED, etc.
  details           JSONB,                      -- stack trace, raw query, etc.
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_audit_log_created_at ON audit_log (created_at DESC);
CREATE INDEX IF NOT EXISTS idx_audit_log_user_timestamp ON audit_log (user_id, timestamp);
CREATE INDEX IF NOT EXISTS idx_audit_log_trace_id ON audit_log (trace_id);
```

**Grant model:**
- `app_user` role: SELECT and INSERT only (no UPDATE/DELETE). Allows audit logging writes in application code.
- Revoke UPDATE/DELETE from all roles including postgres superuser inside production contexts (via trigger rejection).

### Tamper-Proof Enforcement Mechanism

Implemented at both the database and application levels. See tasks 2.1-2.5 for the full implementation details. Key design points:

- **Database level:** `UPDATE`/`DELETE` grants on `audit_log` and `audit_log_tamper_attempt_log` are revoked from `app_user`. A `BEFORE UPDATE OR DELETE` trigger on both tables raises an exception for any modification attempt. The trigger name is `audit_log_tamper_protection`.
- **Application level:** `src/middleware/audit-tamper-guard.ts` catches database trigger exceptions and writes the attempt to `audit_log_tamper_attempt_log` via `logTamperAttempt()`. This provides a secondary fail-safe if the trigger is somehow bypassed and documents all tampering attempts for the auditor.

### Audit Log Query and Integrity Verification

**Query endpoint** (`GET /api/v1/audit/log`):
- Parameters: `start_date` (ISO 8601), `end_date` (ISO 8601), `user_id` (optional UUID), `limit` (default 100, max 1000), `cursor` (for pagination).
- Authorization: `role: 'auditor'` or `'finance_head'`, no location scope (auditors can see all locations).
- Response:
  ```json
  {
    "entries": [
      {
        "log_id": "...",
        "trace_id": "...",
        "user_id": "...",
        "role": "...",
        "location_id": "...",
        "timestamp": "2026-07-13T14:30:45.123Z",
        "endpoint": "/api/v1/events",
        "method": "POST",
        "event_id": "...",
        "http_status": 201,
        "error_code": null
      }
      // ... more entries
    ],
    "range_digest": "sha256_hash_of_log_ids",
    "sequence_check": {
      "is_contiguous": true,
      "first_log_id": "...",
      "last_log_id": "...",
      "gap_count": 0
    },
    "archived_entries_count": 0,
    "next_cursor": "..." // pagination token
  }
  ```
- **Range digest:** SHA-256 hash of the concatenated `log_id` values in the response, in order. This allows an offline auditor to verify that no entries were inserted, deleted, or reordered without re-querying.
- **Sequence check:** verify that the returned entries have no gaps in `log_id` ordering (if `max(log_id) - min(log_id) + 1 != count`, there is a gap). Return metadata about any gaps.

### Retention and Archival Strategy

- **Online retention:** 8 financial years (default). Audit entries older than 8 years are eligible for archival.
- **Archival process:** A scheduled CLI job (`src/cli/archive-audit-log.ts`, run nightly or weekly) exports entries older than the retention threshold to an S3 Glacier archive (or a local tar.gz for dev/test) and marks them as `archived = true`.
- **Recovery:** If an auditor queries a date range that includes archived entries, the endpoint returns `archived_entries_count` and an estimated recovery time (48 hours for Glacier). A separate restoration endpoint or manual process restores entries to queryable state.
- **Protection:** Any attempt to delete or purge entries within the 8-year retention window is rejected with `AUDIT_LOG_TAMPER_ATTEMPT`, whether attempted via API or direct database connection.

### Lifecycle: Disabled Edit Log (AC4)

This is a critical invariant: the edit log cannot be disabled during operation. A misconfigured environment or a malicious actor cannot turn off audit logging.

1. **Startup guard:** If `AUDIT_LOG_ENABLED=false` is detected in the environment at startup, the application fails fast with a critical log message and exits. Do not allow the app to boot with audit logging disabled. Also validate `AUDIT_LOG_RETENTION_YEARS` is a positive integer at startup (fail fast if not).
2. **Runtime config endpoint:** A `PUT /api/v1/config/audit-log-enabled` endpoint (scoped to `system_administrator` role, module `config`) nominally allows toggling the setting, but it is rejected at the application level:
   ```
   IF attempted_value === false THEN
     LOG AUDIT_LOG_TAMPER_ATTEMPT to audit_log_tamper_attempt_log
     REJECT with 423 AUDIT_LOG_DISABLED
   ENDIF
   ```
   The rejection is written to `audit_log_tamper_attempt_log` (not `audit_log`) because no event transaction exists — the write is blocked before `persistEvent` is called.
3. **Pre-flight check:** Every mutating request checks the in-memory `AUDIT_LOG_ENABLED` flag before processing. If `false`, return HTTP 423 `AUDIT_LOG_DISABLED`, write the blocked attempt to `audit_log_tamper_attempt_log`, and skip the event write.

**`http_status` population:** For successful event writes, the audit entry's `http_status` is set to the success status code (201 for POST, 200 for PUT/PATCH) since the INSERT succeeded. Failed event writes roll back the transaction and produce no entry in the main `audit_log` (the transaction is atomic). Tamper attempts and `AUDIT_LOG_DISABLED` rejections are written to `audit_log_tamper_attempt_log` instead, with `error_code` set to the rejection reason.

### SCIM System Events

When Story 1.2's SCIM provisioning/deprovisioning endpoints emit events (`user.provisioned`, `user.deprovisioned`, `user.roles_updated`), they must appear in the audit log with:
- `user_id`: a fixed system-actor UUID (e.g., `00000000-0000-0000-0000-000000000001`, documented in code)
- `role`: `"system"`
- `location_id`: `"*"` (not location-specific)
- All other fields (timestamp, trace_id, endpoint, method, event_id) per the event envelope

This ensures SCIM events are auditable and traceable without conflating them with user-initiated operations.

### Architecture Constraints

1. **Atomic writes:** The audit log entry must be persisted in the same database transaction as the triggering event. If the event write rolls back, the audit entry does not appear. The `persistEvent()` function accepts an optional `AuditContext` parameter (`{ trace_id, user_id, role, location_id, endpoint, method }`) from the API layer; `trace_id` is a per-request UUID generated by the router at the start of each request, NOT extracted from the client-supplied event body.
2. **Tamper attempts use a separate table:** `audit_log_tamper_attempt_log` is written outside the event transaction (via `logTamperAttempt()`) because tamper events and `AUDIT_LOG_DISABLED` rejections occur before or independently of `persistEvent`. This table is also append-only and trigger-protected.
3. **No external dependencies:** S3/Glacier is optional for archival; for local dev/test, archive to a local tar.gz or skip archival entirely.
4. **No caching:** Always read the latest `AUDIT_LOG_ENABLED` flag from the config (no in-memory cache that could stale). Alternatively, cache with a very short TTL (seconds) and accept a few seconds of degraded write latency if disabled during a transition.
5. **Query performance:** The `audit_log` table will grow without bound (by design - it is append-only). Use archival and retention policies to keep the online table manageable (1-3 years of data, ~100M entries at high volume). Index on `created_at DESC` and `(user_id, timestamp)` for common query patterns.

### What This Story Does NOT Include

- Full GDPR/CCPA right-to-be-forgotten compliance (the audit log cannot be deleted by the data subject; it is immutable by design).
- Encryption-at-rest for the audit log (PostgreSQL's own TLS and access controls are assumed sufficient; database-level encryption is an infrastructure concern, not a story scope).
- Replication of the audit log to a separate read-only audit database (this story assumes a single primary PostgreSQL instance; geo-replication is out of scope).
- External SIEM integration (the audit log is queryable via API; downstream tools can consume it via scheduled exports).
- Automatic expiration of archived entries (the 8-year retention is enforced by rejecting deletes, not by automatic purging).
- Custom audit event types beyond the standard request/response envelope (future module stories may emit domain-specific audit events, but they all go into the same table).

### Review Findings

Adversarial code review on 2026-07-14 (uncommitted changes; layers: Blind Hunter, Edge Case Hunter, Acceptance Auditor). Severity in brackets reflects consequence for this compliance-critical system.

Decision needed (resolve before patching):

- [ ] `[Review][Decision]` [HIGH] Sequence gap detection is mathematically meaningless with random UUID `log_id` — `is_contiguous`/`gap_count` in the auditor query response is computed by comparing `gen_random_uuid()` (v4, random) values as if they were monotonic. This will produce false gap reports essentially at random and gives auditors a false sense of tamper-evidence (AC3, Task 4.4). Decide: add a monotonic `BIGSERIAL seq_no` column to `audit_log` for real gap detection, or drop the `is_contiguous`/`gap_count` claim from the response and rely solely on `range_digest` (plus DB-level immutability) as the tamper-evidence mechanism. [src/api/v1/audit.ts:46-48, read/projections/audit_log.sql]
- [ ] `[Review][Decision]` [HIGH] `auditConfig.enabled` is hardcoded `true as const` — the pre-flight `if (!auditConfig.enabled)` check in `events.ts` is statically unreachable dead code, and Task 7.5's test (disable audit log, verify writes blocked with 423) is impossible to execute since the app refuses to *start* rather than serving requests when `AUDIT_LOG_ENABLED=false`. AC4 describes "a configuration flag attempts to disable the edit log" as a runtime event, not a boot-time one. Decide: keep audit logging startup-immutable (arguably more secure; drop the dead runtime check and rewrite Task 7.5 as a startup-failure test) or make it live-toggleable (e.g., a DB-backed flag re-read per request) so AC4's runtime-block behavior is actually exercised. [src/config/audit.ts, src/api/v1/events.ts:42-45]

Patch (unambiguous fixes):

- [ ] `[Review][Patch]` [HIGH] SCIM-emitted events (`user.provisioned`, `user.roles_updated`, `user.deprovisioned`, `user.reactivated`) bypass the audit log entirely — `persistEvent()` is called without an `auditCtx`, so no entry is written despite Task 6.3 requiring a system-actor audit entry for these events [src/adapters/iam/scim.ts:50-61,75-81,95-101,113-119]

Fix: use a composite cursor `(timestamp, log_id) > ($cursor_ts, $cursor_id)` [src/api/v1/audit.ts:30-35]
