# Story 3.5: Directed Putaway and Location Override Recording (UJ-PUT-01, FR-W-03)

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a store assistant (role `store_assistant`),
I want the system to direct me to the best bin for each received lot and let me scan the actual bin I used, recording any override as an authoritative correction event with a reason code,
so that every physical location is reflected in the system, my real-world knowledge improves the directed suggestions for the whole team, and last-writer-wins is never applied to location.

**Scope note:** this story builds the velocity-classification capability (ABC classes derived from putaway/override frequency) and the override-driven re-slotting job from scratch. Neither exists yet; do not search for pre-existing infrastructure.

## Acceptance Criteria

1. **Given** a `ready` putaway task exists for 50 kg of `RM-0042` in `ZONE-AMBIENT` (created by Story 3.4's receiving flow)
   **When** the store assistant opens the task on the edge PWA (`GET` the task's directed suggestion)
   **Then** the system returns a directed bin suggestion (e.g., `BIN-A43`) computed from velocity class, item size class against bin size class, zone rules, and current occupancy (FR-W-03).

2. **Given** a received lot carries size class `LARGE` and `BIN-A43` carries size class `SMALL` (size attributes from Story 3.1 `location_register`)
   **When** the directed putaway suggestion is computed
   **Then** `BIN-A43` is excluded and the suggestion returns the nearest eligible bin whose size class fits the lot (FR-W-03 size criterion); if no eligible bin exists, the response says so explicitly rather than returning an invalid suggestion.

3. **Given** the store assistant scans `BIN-A47` instead of the suggested `BIN-A43`
   **When** the scan is confirmed with reason code `"better_space_available"` and confidence `"certain"`
   **Then** a `location.override` event is recorded with the asserted location `BIN-A47`, the expected (directed) location `BIN-A43`, the reason code, the confidence, and the operator identity; `BIN-A47` becomes the authoritative current location for the lot, and the putaway task completes with `actual_location_id` set to `BIN-A47`.

4. **Given** multiple override events for the same SKU cluster on the same actual bin within a rolling 30-day window
   **When** the re-slotting job built in this story runs (velocity classification plus override-cluster analysis)
   **Then** the directed putaway suggestion for that SKU is updated to reflect the operators' preferred bin, the override improves the suggestion for the whole team (NFR-ADOPT-01), and the job's outcome is queryable (which SKUs were re-slotted, old vs. new suggested bin).

5. **Given** the store assistant is offline when completing the putaway
   **When** the override event is synced
   **Then** it replays in the correct sequence relative to other events for the same lot (client-supplied idempotency keys), and the location projection is updated exactly once — no double-application on retry.
