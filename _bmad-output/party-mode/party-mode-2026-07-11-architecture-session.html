The Paradigm

🏗️
Winston
This isn't a monolith. It's not microservices either. It's a partitioned system with a local-first write path at every site, governed by a central control plane. Every site — plant, warehouse, R&D store, hub — runs its own instance that works when the network doesn't. The central plane owns reconciliation, not operation. The frontline edge doesn't ask permission to capture a gate event.
🧭
John
So the edge owns writes, the center owns truth. What's the latency budget on that reconciliation?
🏗️
Winston
5 seconds ceiling for cross-location sync. Most events — gate entry, weighbridge — can be seconds behind. The ones that can't: stock allocation, IRN-gated dispatch, the QC lockout. Those resolve at the central plane.

AD-1 Partitioned Local-First Paradigm

Binds: all modules, all edge devices
Prevents: a monolith that requires network connectivity for frontline operations
Rule: every site operates its own local-first write path. The edge device writes to local SQLite, not the central API. The degraded state is visible on the device.

The Event Model

📋
Mary
If the gate event is captured on the device and the central plane hasn't seen it yet, where does the statutory edit log live? FR-AC-13 says tamper-proof, non-disableable, auditor-reportable. When the auditor opens the book, which one is the book?
🏗️
Winston
The device event is the source event. It carries a device-assigned sequence number, a hash of the prior event, and a device signature. When it syncs to the central plane, the central plane appends it to the global event stream. The auditor opens the central plane. The device ledger is the evidence, not the book.
🦺
Ravi
The guard at 2am captures a gate entry. Five minutes later, the shift supervisor sees an alert on the dashboard: 'gate event pending sync — 1047, Truck MH-04-EX-8821, 02:14'. If the supervisor doesn't see the pending event, he'll call the guard and ask why the truck isn't in the system.
🏗️
Winston
Exactly. The degraded state is visible at both ends. Nobody calls Suresh.

AD-2 Gate-Token Event Chain

Binds: gate, weighbridge, receiving, putaway
Prevents: disconnected gate and weighbridge events that cannot be correlated
Rule: every inbound event chain starts with a gate event creating a vehicle-to-PO binding token. Events are stitched by token, not by timestamp.

AD-16 Idempotency Keys on All Edge-Originated Commands

Binds: all edge devices, central event store
Prevents: duplicate events from multiple devices capturing the same physical fact
Rule: every command carries an idempotency_key. Central plane deduplicates. HTTP 409 on duplicate.

The Stack

🧭
John
What's the event store? Because if we're building a custom platform on a 36-month timeline, I don't want us also building an event store.
🏗️
Winston
PostgreSQL 18.4 with the event-sourcing-lite pattern. One domain_events table, 8 columns, per-stream optimistic concurrency. One database to operate, backup, and monitor. KurrentDB 26.1 is the mature alternative if we want built-in projections, but PostgreSQL can handle our scale.
🦺
Ravi
One database. One backup, one restore, one thing to wake up at 3am. I've seen too many 'purpose-built' services that became the thing nobody knew how to fix when the consultant left.
🎨
Sally
What actually runs on the device? The edge devices are rugged tablets and scanners. They need to work offline.
🏗️
Winston
PWA with a service worker and local SQLite. PowerSync handles the sync — PostgreSQL to on-device SQLite, upload queue, crash recovery. For the framework: Next.js 16 or TanStack Start 1.x. The sync layer is abstracted from the framework.

Stack: PostgreSQL 18.4 · Node.js 24 LTS · PowerSync 1.23.x · Next.js 16 / TanStack Start 1.x · AWS Mumbai + Hyderabad

Deployment: ECS Fargate (no Kubernetes), RDS Aurora PostgreSQL Multi-AZ, CloudFront CDN. Three environments. DR via Aurora Global Database with RTO 4h, RPO 1h.

The Key Trade-Offs

🧭
John
Which five capabilities for first go-live?
🏗️
Winston
Compliance spine, core inventory, frontline gate edge, job-work services, and R&D/maker-hub. The spine-first approach means the compliance layer is built and acceptance-tested before any module lands.
📋
Mary
The compliance spine maps to 27 requirements. The spine acceptance contract is five testable contracts: edit log integrity, DOA registry resolution, event-sourced location, calibration lockout, business-stream tagging. That's the gate for Phase 1.
🦺
Ravi
The pilot site should be the ugliest one. The worst connectivity, the most paper registers, the most tribal knowledge. If the spine survives that site, it survives anywhere.

Winston's Closing

🏗️
Winston
The spine is consistent. Every AD prevents a specific divergence. The paradigm — partitioned local-first — is the load-bearing decision. Everything else flows from it. The stack is PostgreSQL 18.4, Node.js 24 LTS, PowerSync for sync, Next.js 16 or TanStack Start 1.x for the frontend, AWS Mumbai/Hyderabad for deployment. The event envelope is uniform across the entire system. The dependency direction is strict — compliance spine at the bottom, no circular dependencies. The data ownership is clear — one owner per entity, read models are shared. The testing strategy is concrete — five acceptance contracts that prove the spine works before any module lands. I approve the spine.
