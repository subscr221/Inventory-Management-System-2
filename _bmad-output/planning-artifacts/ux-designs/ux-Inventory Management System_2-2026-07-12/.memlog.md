### 2026-07-12 | Design Philosophy | Decision: Frontline-First Over Office-First

**Topic:** Interaction model and primary audience

**Input:**
- Frontline users (gate, weighbridge, putaway, hub) work offline, on tablets, gloved, one-handed
- Administrative users (procurement, finance) work on desktop
- Conflicting UX constraints: speed on tablet vs. power on desktop

**Decision:** 
- Primary: Tablet (7–10", landscape, 44×44px touch targets)
- Secondary: Desktop (1280px+, multi-column, keyboard shortcuts)
- Interaction model: Scan-first, forms-secondary, glove-friendly

**Rationale:**
- Revenue exposure drives frontline: gate and warehouse flows are highest-frequency, lowest-tolerance (2am captures, offline data, zero-loss requirement)
- Frontline culture is scan-first (barcode labeling assumed deployed); typing is fallback
- Glove and one-handed operation are stated requirements (warehouse reality)

**Outcomes:**
- DESIGN.md sections 4.1–4.2 define responsive grid and tablet-primary layout
- EXPERIENCE.md sections 4.1 defines scan input pattern
- Accessibility baseline elevated (WCAG 2.1 AA non-negotiable; scan-first requires alt paths)

### 2026-07-12 | Component Patterns | Decision: Stateless Until Offline-Aware

**Topic:** How UI components signal state and handle offline/sync

**Input:**
- PowerSync event store means state is bidirectional (local queue, central sync)
- Offline-first principle: device queues events locally; no "error" until sync fails
- Four states: ONLINE (synced), OFFLINE (captured), SYNCING (in progress), SYNC ERROR (failed)
- Approval workflows must be reversible (edit log immutability means reversals, not edits)

**Decision:**
- Every screen has a state machine (5.1) with 4 states visible in a footer badge
- Scan input always autofocuses; recent scans shown below (quick-tap reuse)
- Approval cards show decision-maker, reason field, and [Confirm Reversal] button
- Locator override surfaces mismatch, collects reason code and confidence

**Rationale:**
- State machine prevents "silent success" (device queued offline but never synced)
- Autofocus on scan field speeds frontline moment-of-use (no navigation needed)
- Approval + reversal path meets "immutable, auditable" requirement (no edits, only reversals)
- Locator override with reason feeds ABC algorithm learning (NFR-ADOPT-01: visible value loop)

**Outcomes:**
- EXPERIENCE.md sections 4.1–4.4 define component patterns
- EXPERIENCE.md section 5 defines state machines (offline, approval, QC)

### 2026-07-12 | Accessibility | Decision: WCAG 2.1 AA as Baseline, Extended for Frontline

**Topic:** Accessibility standards and frontline-specific needs

**Input:**
- PRD requires WCAG 2.1 AA
- Frontline users are diverse: varying ages, literacy, motor ability
- Warehouse environments are loud, bright/dark, gloved, one-handed
- Compliance requirement: edit log, audit trail, immutability (all accessibility adjacent)

**Decision:**
- WCAG 2.1 AA as baseline (7:1 AAA preferred for body text)
- Extended frontline-specific accessibility:
  - 44×44px touch targets (tactile; glove-friendly)
  - Glove spacing (8px minimum between targets)
  - No time limits on forms (except 30-sec scan timeout with restart notification)
  - Screen reader support (labels, aria-live, alt text)
  - Keyboard navigation (Tab, Shift+Tab, Enter, Escape, Alt+shortcuts)
  - Dyslexia-friendly typography (1.5 line-height, 0.02em letter-spacing)

**Rationale:**
- Warehouses employ diverse populations; accessibility is inclusion, not compliance checkbox
- Glove-friendly UX is not "nice to have" for warehouse; it's essential
- Motor control variance means no small gestures; large targets and spacing are essential
- Cognitive accessibility (time limits, jargon, clear error messages) is as important as visual

**Outcomes:**
- DESIGN.md section 9 specifies WCAG 2.1 AA with extended frontline considerations
- EXPERIENCE.md section 7 details keyboard navigation, screen reader support, motor/cognitive accessibility
- Assumption tagged: [ASSUMPTION] Gauge calibration lockout (FR-M-13) enforced by backend; UX shows status only

### 2026-07-12 | Information Architecture | Decision: Role-Based Dashboards, Not Monolithic Navigation

**Topic:** How to structure navigation for 6 frontline + 3 supervisory + 3 admin roles

**Input:**
- System serves 12 distinct roles (source §11)
- Frontline roles see task queues (gate, receiving, putaway, hub)
- Supervisory roles approve/release/inspect
- Admin roles configure and audit
- All roles need fast access to their primary workflow

**Decision:**
- Tab-based navigation (left sidebar desktop, bottom bar mobile) with 5–7 primary categories
- Role-based dashboard showing pending tasks (approvals, QC, release gates)
- Deep linking to task-specific flows (e.g., gate event detail → receiving form)
- Role provisioning via SCIM (INT-IAM-02); role changes reflect in IA immediately (no UI change needed)

**Rationale:**
- Role-scoped IA reduces cognitive load (each role sees only what they need)
- Task queues (pending approvals, QC holds, indents) are the "inbox" for each role
- Deep linking allows email/SMS/push to jump directly to action (no navigation chains)
- SCIM provisioning means IA updates without code change (federated access)

### 2026-07-12 | Offline-First State Machine | Decision: Explicit "Captured, Pending Sync" Over Implicit

**Topic:** How to communicate offline state to frontline users

**Input:**
- Architecture requires offline-first (PowerSync, edge PWA)
- NFR-ADOPT-01 demands visible feedback ("captured frontline knowledge must visibly benefit")
- Silent failures are unacceptable (data loss or invisible queuing kills adoption)
- Users must understand: "I'm offline, data is safe, sync will happen"

**Decision:**
- Footer badge in every screen showing sync state (persistent, always visible)
- Four explicit states: ONLINE, OFFLINE (Captured), SYNCING, SYNC ERROR
- Badge includes icon + label + [Retry Sync] button (SYNC ERROR only)
- No jargon: "Captured, pending sync" not "offline mode" or "queued events"

**Rationale:**
- Persistent footer is glanceable (not modal; not intrusive)
- Explicit states prevent surprise ("where's my data?")
- [Retry Sync] button empowers user recovery (not "try again later")
- Label text educates (new users learn what "captured" means)

**Outcomes:**
- EXPERIENCE.md section 5.1 defines state machine with UI indicators
- DESIGN.md section 2.3 specifies sync-state badge colors and layout
- Assumption tagged: [ASSUMPTION] PowerSync client configuration deferred to backend (architecture scope)

### 2026-07-12 | Compliance & Reversals | Decision: No Deletes, Only Reversals (Edit Log Compliance)

**Topic:** How to implement audit-trail requirement without compromising UX

**Input:**
- FR-AC-13 (statutory edit log) requires tamper-proof, immutable audit trail
- Regulatory requirement: every mutation is logged; no deletion; reversals are appended
- UX challenge: users expect "undo"; system requires "create reversal event"
- Companies Act 2013 audit-trail proviso: nothing is deleted

**Decision:**
- No UI delete buttons; all actions have a reversal path
- Reversal is a new event with actor, timestamp, reason (fully auditable)
- Reversal reason is collected via modal (e.g., "Wrong vehicle" for gate event reversal)
- Cancelled orders, rejected indents, reversed captures all create explicit events
- Edit log shows both the original action and the reversal with timeline

**Rationale:**
- Immutable events satisfy regulatory requirement (no hidden edits or deletions)
- Reversal reason provides audit context ("why was this undone?")
- Frontend never sees "delete"; backend posts "reversal" event
- User experience is preserved (they get the undo they expect; system gets the audit trail it needs)

**Outcomes:**
- EXPERIENCE.md section 6.3 defines reversal pattern with modal and reason collection
- DESIGN.md section 8 (Do's/Don'ts) includes: "✓ Provide undo; ✗ Don't delete"
- Assumption tagged: [ASSUMPTION] Edit log backend prevents reversal events themselves from being modified (immutable store)

### 2026-07-12 | Microcopy & Voice | Decision: Action-First, No Jargon, Compliance-Informed

**Topic:** What language to use in UI

**Input:**
- Users are warehouse staff (varied literacy, domain expertise)
- Compliance language (edit log, audit trail, immutability) is foreign to frontline
- Need to bridge: regulatory precision without jargon
- Example contrast: "Dispatch blocked: QC Hold on lot QZ-4421" (compliance) vs. "We're waiting for quality approval" (accessible)

**Decision:**
- Action-first labels: "Approve Indent" not "Approval".
- No jargon unless domain-specific: "warehouse block" OK, "inbound vehicular transaction" not OK.
- Errors are actionable: "Weight out of tolerance (±5%) — enter again or call supervisor".
- Compliance language is embedded but not exposed: "QC Hold" is the domain term; system uses it.
- Confirmations are explicit: "Indent approved, submitted to procurement. Status updates via push notification."

**Rationale:**
- Action-first language is faster to scan (gate officer reading under time pressure)
- No jargon reduces cognitive load (warehouse staff don't need to learn ERP terminology)
- Actionable errors reduce support load (users know what to do next)
- Compliance-informed language is precise but accessible (domain terms are explained, not hidden)

**Outcomes:**
- EXPERIENCE.md section 3 defines voice and tone with examples
- DESIGN.md section 8 (Do's/Don'ts) includes microcopy guidance
- Assumption tagged: [ASSUMPTION] Localized language (Hindi, Tamil, Telugu, Kannada) deferred to Phase 2
